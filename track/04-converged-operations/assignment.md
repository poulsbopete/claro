# Lab 4: Converged Operations — SIEM Value Without Double Ingest

## The Splunk Enterprise Security Tax

To get security detection on application logs in Splunk, you need:

1. **Splunk Cloud** — your observability platform (logs, metrics)
2. **Splunk Enterprise Security (ES)** — a separate product, separate license, separate management overhead
3. **CIM Normalization** — your logs must be mapped to Splunk's Common Information Model before ES can process them
4. **Double Ingest** — data typically needs to flow into both Splunk Cloud (for operational queries) and Splunk ES (for security detection) — you pay to ingest the same data twice

The result: SIEM coverage on application logs costs approximately **2× the ingest cost**, plus an additional ES license, plus CIM mapping work.

**In Elastic, observability data is security data.** The same Elasticsearch index that your APM dashboards and log queries read from can be targeted by Elastic Security detection rules, machine learning anomaly jobs, and alert actions — with no additional ingest pipeline, no data duplication, no second license required.

## Scenario: A Brute-Force Attack in Your Application Logs

The setup script injected a realistic brute-force attack pattern into the `lab1-app-logs-*` index — the **same index you used for APM correlation in Lab 1**:

- **15 authentication failures** from IP `198.51.100.42` in a 5-minute window
- **1 successful login** at the end — indicating the brute force succeeded
- **Target user:** `admin`
- **Targeted endpoint:** `/api/auth/login` on `checkout-service`

A detection rule was pre-created in Kibana Security targeting this index pattern.

---

## Step 1: Confirm the Attack Events Are in Your Observability Index

```bash
source /etc/profile.d/lab4_env.sh

# Find the attack events — same index as Lab 1 APM logs
curl -sk -X POST "${ES_URL}/_query" \
  -H "Authorization: ApiKey ${ES_API_KEY}" \
  -H "Content-Type: application/json" \
  -d "{
    \"query\": \"FROM ${LAB4_LOG_INDEX} | WHERE \\\"source.ip\\\" == \\\"${LAB4_ATTACK_IP}\\\" | KEEP @timestamp, log.level, message, event.action, event.outcome, http.response.status_code | SORT @timestamp ASC\"
  }" | jq '.values[] | {ts: .[0], action: .[3], outcome: .[4], status: .[5]}'
```

> **This is the critical insight:** You are querying `lab1-app-logs-*` — the **same observability index** from Lab 1. No separate SIEM ingest. No data duplication. No additional cost.

---

## Step 2: Run the Brute-Force Detection Query Manually

This is the underlying logic of the detection rule — a query you can run in ES|QL:

```bash
curl -sk -X POST "${ES_URL}/_query" \
  -H "Authorization: ApiKey ${ES_API_KEY}" \
  -H "Content-Type: application/json" \
  -d "{
    \"query\": \"FROM ${LAB4_LOG_INDEX} | WHERE @timestamp > NOW() - 10 minutes AND http.response.status_code == 401 | STATS failure_count = COUNT(), unique_endpoints = COUNT_DISTINCT(url.path) BY \\\"source.ip\\\", user.name | WHERE failure_count >= 5 | SORT failure_count DESC\"
  }" | jq '.values[] | {source_ip: .[2], user: .[3], failures: .[0], endpoints_hit: .[1]}'
```

## Step 4: View the Generated Alert

1. In Kibana, navigate to **Security → Alerts**
2. You should see an alert generated by the brute-force rule
3. Click the alert to expand it
4. In the alert detail panel, note:
   - The source index: `lab1-app-logs-*` (your observability index)
   - The MITRE ATT&CK mapping: `T1110 - Brute Force`
   - The risk score and severity

> In Splunk ES, this alert would reference data from a **separate CIM-normalized datastore**. In Elastic, the alert **points directly to the original log document** in your observability index.

---

## Step 5: Pivot from Alert to Original Log Context

In Kibana, from the alert detail view:

1. Click **Investigate in Timeline**
2. You are now in Elastic Security's Timeline view, showing the raw log events
3. Click on any event → **View details**
4. Note the `trace.id` field — this is the **same field** you used in Lab 1 to correlate to APM

> In Elastic, a security alert on an app log can be correlated back to its APM trace **in the same platform**. In Splunk, your security analyst would need to copy a log entry to Splunk Cloud to find the associated APM data — if it is even available.

## Appendix: Manually Create the Detection Rule (if not auto-created)

If the rule was not automatically created, create it via the Kibana API:

```bash
source /etc/profile.d/lab4_env.sh

curl -sk -X POST "${KIBANA_URL}/api/detection_engine/rules" \
  -H "Authorization: ApiKey ${ES_API_KEY}" \
  -H "Content-Type: application/json" \
  -H "kbn-xsrf: true" \
  -d '{
    "type": "query",
    "language": "kuery",
    "name": "[Lab 4] Brute Force: Multiple Auth Failures",
    "description": "Detects repeated auth failures from same IP on observability logs",
    "severity": "high",
    "risk_score": 73,
    "enabled": true,
    "index": ["lab1-app-logs-*"],
    "query": "http.response.status_code: 401 AND event.action: authentication_failure",
    "from": "now-10m",
    "interval": "5m",
    "rule_id": "lab4-brute-force-manual-001",
    "threat": [{
      "framework": "MITRE ATT&CK",
      "tactic": {"id": "TA0006", "name": "Credential Access", "reference": "https://attack.mitre.org/tactics/TA0006/"},
      "technique": [{"id": "T1110", "name": "Brute Force", "reference": "https://attack.mitre.org/techniques/T1110/"}]
    }],
    "actions": []
  }' | jq '{id: .id, name: .name, enabled: .enabled}'
```

## Competitive Takeaway

| Capability | Elastic | Splunk ES |
|------------|---------|-----------|
| Detection rules on observability data | ✅ Same index | ❌ Separate product |
| Additional ingest for SIEM coverage | ✅ Zero | ❌ Full double-ingest |
| CIM/normalization required | ✅ None (ECS native) | ❌ CIM mapping required |
| Alert → trace correlation | ✅ Same platform | ❌ Cross-product navigation |
| Single license for obs + security | ✅ | ❌ Cloud + ES licenses |
| MITRE ATT&CK alignment built-in | ✅ | ✅ (ES only) |

> **The Splunk model creates a financial disincentive to security coverage.** Every log that could be a security signal costs twice as much to protect. Elastic's converged architecture means there is no trade-off between observability depth and security breadth.
